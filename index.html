01 — Foundations

What is a Transaction?

A transaction is a sequence of operations that executes as a single logical unit — all succeed, or all are rolled back.

📝
BEGIN
TX started
→
💸
Debit $100
Account A
→
💳
Credit $100
Account B
→
✅
COMMIT
Persisted
SQL
BEGIN TRANSACTION;
  UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
  UPDATE accounts SET balance = balance + 100 WHERE id = 'B';
COMMIT;

-- If any step fails, ROLLBACK undoes everything
02 — Core Properties

ACID Properties

The four guarantees that make database transactions reliable.

A
Atomicity
All or Nothing
Every operation in a transaction either completes fully or the entire transaction is rolled back. Partial updates never persist.
Debit fails → Credit never happens
C
Consistency
Rules always hold
A transaction takes the database from one valid state to another. Constraints, triggers, and rules are always enforced.
balance ≥ 0 constraint always satisfied
I
Isolation
Concurrent isolation
Concurrent transactions execute as if they were serial. Intermediate states of a transaction are not visible to others.
Two users can't both claim the last seat
D
Durability
Survives crashes
Once committed, data is permanently stored even if the system crashes. Achieved via Write-Ahead Logging (WAL).
Power cut after COMMIT → data still there

Trade-off between data consistency and concurrent performance. Stronger isolation = less throughput.

04 — Microservices

The Distributed Transaction Problem

In microservices, each service owns its database. A single business operation spans multiple services — there's no global transaction manager.

📦
Create Order
Order DB ✓
→
💳
Charge Payment
Payment DB ✓
→
📉
Reduce Stock
Inventory DB ✗
→
💥
Inconsistent!
Paid, no stock
⚠️ Payment was charged but inventory was never reduced. Without distributed coordination, the system is left in an inconsistent state — traditional ROLLBACK doesn't span service boundaries.
05 — Solutions

Distributed Transaction Patterns

Proven strategies to maintain consistency across service boundaries.

Pattern 01
Two-Phase Commit (2PC)
Prepare → Commit/Abort
A coordinator asks all participants to prepare locks, then issues a final commit or abort. Strong consistency but synchronous and blocking.
Phase 1: Coordinator → "Can commit?"
Phase 2: All say yes → Commit!
Any say no → Abort!
Strong consistency guarantee
Coordinator crash = participants stuck
High latency from network round-trips
Pattern 02
SAGA Pattern
Local TXs + Compensations
Break into a sequence of local transactions. On failure, execute compensating transactions to undo prior steps.
OrderCreated → PaymentDone
→ StockFailed
→ RefundPayment (compensate)
→ CancelOrder (compensate)
No distributed locks needed
Resilient and scalable
Eventual consistency only
Pattern 03
Outbox Pattern
Transactional messaging
Write events to an outbox table in the same local transaction as business data. A relay process then publishes to the message broker.
BEGIN TX;
INSERT orders ...
INSERT outbox (event) ...
COMMIT; // atomic!
// Relay reads outbox → broker
Exactly-once delivery guarantee
No dual-write problem
Extra table + relay process
Pattern 04
Event Sourcing
Store events, not state
Instead of persisting current state, store an immutable log of events. Rebuild state by replaying the event stream.
OrderCreated { id: 1 }
PaymentCharged { amt: 99 }
ItemShipped { track: XYZ }
State = replay(events)
Complete audit trail
Time-travel queries
Complex querying / projections

06 — Theory

CAP Theorem

In any distributed system, you can only guarantee 2 of these 3 properties simultaneously.

CP
Consistent + Partition Tolerant
Sacrifices availability during network partitions. Returns an error rather than stale data.
HBase · Zookeeper · etcd · MongoDB (strong)
AP
Available + Partition Tolerant
Always responds but may return stale data. Eventually converges to consistency.
Cassandra · CouchDB · DynamoDB · Riak
CA
Consistent + Available
Only possible on a single node. Cannot tolerate network partitions — not truly distributed.
Single-node RDBMS · PostgreSQL (standalone)
💡 In practice, partitions always happen in distributed systems. The real choice is between CP (refuse stale reads) and AP (serve possibly stale data). Choose based on your domain's tolerance for inconsistency.
07 — Summary

Local vs Distributed Transactions

Key differences between single-database and distributed transaction management.

Property Local DB Transaction Distributed Transaction
Consistency model Strong (ACID) Eventual (BASE)
Rollback Instant, built-in Compensating transactions
Coordinator Single DB engine SAGA orchestrator / events
Performance Fast (no network) Slower (network hops)
Locking Row/page-level locks No global locks (SAGA)
Failure handling ROLLBACK Idempotency + compensation
Best patterns BEGIN / COMMIT / ROLLBACK SAGA · Outbox · Event Sourcing
